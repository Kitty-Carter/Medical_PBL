# 常见问题（FAQ）

## 部署相关

### Q1：部署后访问页面显示 502 Bad Gateway

**原因**：后端服务未正常启动或 Nginx 反向代理配置错误。

**解决方案**：
1. 检查后端服务状态：`pm2 status` 或 `systemctl status medical-pbl`
2. 查看后端日志：`pm2 logs medical-pbl` 或 `journalctl -u medical-pbl -f`
3. 确认 Nginx 配置中 `proxy_pass` 指向正确的后端端口（默认 3000）
4. 重启服务：`pm2 restart medical-pbl`

### Q2：数据库连接失败，提示 "connect ECONNREFUSED"

**原因**：MySQL 服务未启动、连接信息配置错误或防火墙阻止。

**解决方案**：
1. 检查 MySQL 服务状态：`systemctl status mysql`
2. 验证 `.env` 中的数据库连接信息（`DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`）
3. 测试数据库连接：`mysql -h 127.0.0.1 -u root -p`
4. 检查防火墙是否放行 MySQL 端口（默认 3306）

### Q3：文件上传失败或上传后无法访问

**原因**：`uploads/` 目录权限不足或磁盘空间不足。

**解决方案**：
1. 检查目录权限：`ls -la uploads/`，确保 Web 服务用户有读写权限
2. 修复权限：`chmod -R 755 uploads/ && chown -R www-data:www-data uploads/`
3. 检查磁盘空间：`df -h`
4. 确认 Nginx 配置中 `client_max_body_size` 足够大

### Q4：HTTPS 证书配置后浏览器提示不安全

**原因**：证书过期、证书链不完整或配置错误。

**解决方案**：
1. 检查证书有效期：`openssl x509 -in /path/to/cert.pem -noout -dates`
2. 使用 Let's Encrypt 自动续期：`certbot renew --dry-run`
3. 确认 Nginx 配置中同时包含完整证书链和私钥
4. 重启 Nginx：`systemctl reload nginx`

## AI 功能相关

### Q5：AI 助手无响应或返回错误

**原因**：API Key 无效、余额不足、网络不通或模型配置错误。

**解决方案**：
1. 检查 `.env` 中 `DEEPSEEK_API_KEY` 是否正确
2. 测试 API 连通性：
   ```bash
   curl -X POST https://api.deepseek.com/v1/chat/completions \
     -H "Authorization: Bearer $DEEPSEEK_API_KEY" \
     -H "Content-Type: application/json" \
     -d '{"model":"deepseek-chat","messages":[{"role":"user","content":"test"}]}'
   ```
3. 检查 API 余额：登录 DeepSeek 平台查看
4. 确认服务器能访问外网：`curl -I https://api.deepseek.com`

### Q6：AI 回复速度很慢

**原因**：网络延迟、模型负载高或请求内容过长。

**解决方案**：
1. 检查服务器到 DeepSeek API 的网络延迟：`ping api.deepseek.com`
2. 减少单次请求的上下文长度（调整 `MAX_CONTEXT_LENGTH`）
3. 考虑使用更快的模型（如 `deepseek-chat` 替代 `deepseek-reasoner`）
4. 检查服务器 CPU/内存使用率：`top -bn1 | head -20`

### Q7：AI 评分结果不准确

**原因**：评分提示词（Prompt）不够精确或模型理解偏差。

**解决方案**：
1. 检查 `server.js` 中的评分提示词模板
2. 调整 `TEMPERATURE` 参数（降低可提高一致性，默认 0.3）
3. 在评分逻辑中增加规则校验层
4. 查看 AI 返回的原始内容进行调试

## 性能相关

### Q8：系统响应变慢，CPU 或内存占用高

**原因**：并发请求过多、内存泄漏或数据库查询效率低。

**解决方案**：
1. 查看资源占用：`htop` 或 `top -bn1`
2. 检查 Node.js 进程内存：`pm2 monit`
3. 优化数据库查询，添加索引
4. 增加 PM2 实例数（集群模式）：`pm2 scale medical-pbl 2`
5. 配置 Nginx 限流和缓存

### Q9：数据库查询慢

**原因**：缺少索引、数据量大或查询语句未优化。

**解决方案**：
1. 开启 MySQL 慢查询日志：`SET GLOBAL slow_query_log = 'ON';`
2. 分析慢查询：`mysqldumpslow /var/log/mysql/slow.log`
3. 为常用查询字段添加索引
4. 定期清理历史数据（如 `records/` 中的旧记录）

## 安全相关

### Q10：如何修改默认管理员密码

**解决方案**：
1. 编辑 `.env` 文件，修改 `TEACHER_PASSWORD` 的值
2. 重启服务使配置生效：`pm2 restart medical-pbl`
3. 建议使用强密码（至少 12 位，包含大小写字母、数字和特殊字符）

### Q11：如何防止暴力破解

**解决方案**：
1. 使用强密码策略
2. 配置 Nginx 限流：
   ```nginx
   limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;
   location /login {
       limit_req zone=login burst=3 nodelay;
   }
   ```
3. 启用 fail2ban 防护
4. 定期检查登录日志

### Q12：发现安全漏洞如何报告

请发送邮件至项目维护者邮箱，或提交 GitHub Issue（标记为 Security）。我们将在 48 小时内响应。

## 其他

### Q13：如何备份和恢复数据

**备份**：
```bash
# 备份数据库
mysqldump -u root -p medical_pbl > backup_$(date +%Y%m%d).sql

# 备份上传文件
tar -czf uploads_backup_$(date +%Y%m%d).tar.gz uploads/

# 备份配置文件
cp .env .env.backup.$(date +%Y%m%d)
```

**恢复**：
```bash
# 恢复数据库
mysql -u root -p medical_pbl < backup_20260101.sql

# 恢复上传文件
tar -xzf uploads_backup_20260101.tar.gz
```

### Q14：如何升级到新版本

1. 备份当前数据和配置
2. 下载新版本发布包
3. 解压并覆盖文件（保留 `.env` 和 `uploads/`）
4. 安装新依赖：`npm install --production`
5. 执行数据库迁移（如有）：`npm run migrate`
6. 重启服务：`pm2 restart medical-pbl`
