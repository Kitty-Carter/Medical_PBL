# AI 配置说明

## 概述

Medical PBL 采用 DeepSeek API 驱动 AI 对话，实现"像 DeepSeek 网页一样自然"的多角色课堂互动体验。系统内置三个 AI 角色（A教授、B同学、C同学），通过事件驱动机制自动轮转发言，模拟真实 PBL 课堂讨论。

## 1. API 配置

### 1.1 DeepSeek API（推荐）

在 `.env` 文件中配置：

```bash
# DeepSeek API 密钥（必填）
DEEPSEEK_API_KEY=sk-your-deepseek-api-key-here

# DeepSeek API 地址（可选，默认官方地址）
DEEPSEEK_BASE_URL=https://api.deepseek.com
```

获取 API Key：访问 [DeepSeek 开放平台](https://platform.deepseek.com/) 注册并创建 API Key。

### 1.2 OpenAI API（兼容）

系统同时兼容 OpenAI 格式的 API：

```bash
# OpenAI API 密钥
OPENAI_API_KEY=sk-your-openai-api-key-here

# OpenAI API 地址（可选）
OPENAI_BASE_URL=https://api.openai.com/v1
```

> **注意**：DeepSeek API 和 OpenAI API 二选一即可，系统优先使用 DeepSeek。

## 2. AI 角色体系

系统内置三个 AI 角色，各司其职：

| 角色 | 身份 | 职责 |
|------|------|------|
| **A教授** | 教师/引导者 | 开场定议程、追问深化、阶段总结、纠正跑题、处理危重信号 |
| **B同学** | 同伴学习者 | 降低讨论压力、拆解复杂问题、鼓励发言、提出模糊观点 |
| **C同学** | 医学达人 | 补充医学依据、区分相似概念、引用指南、提醒查证 |

角色规则卡位于 `ai_roles/` 目录：

```
ai_roles/
├── AI_助教团总规则.md      # 角色总体行为规范
├── AI_触发路由规则.md      # 触发条件与角色选择规则
└── AI_兜底回复模板.md      # 异常情况下的兜底回复
```

## 3. 触发路由规则

AI 回复由事件驱动，不是每条学生消息都会触发 AI 发言。

### 3.1 不回复的情况

- 学生打招呼、寒暄
- 学生仅回复"收到""好的""赞同"等简短确认
- 最近 2 条消息中已有 AI 发言
- 学生之间正在连续讨论且无明显冷场
- 当前消息无医学信息也无教学推进价值

### 3.2 高优先级触发事件

| 事件 | 触发条件 | 角色 |
|------|---------|------|
| `teacher_opening` | 新病例开场 | A教授 |
| `ai_paste_suspected` | 检测到 AI 长文粘贴（>500字） | A教授 |
| `red_flag` | 学生提到危重信号 | A教授 |
| `noise_redirect` | 噪声累积（≥2次） | A教授 |

### 3.3 正常轮转事件

| 事件 | 触发条件 | 角色 |
|------|---------|------|
| `student_follow_up` | 学生发言 >50字 | B同学/C同学轮换 |
| `teacher_summary` | 每 8 轮触发一次 | A教授 |
| `rotation` | 默认轮换 | A教授 → B同学 → C同学 |

## 4. 模型参数策略

系统统一使用 `deepseek-chat` 模型，通过参数分档实现不同场景需求：

### 4.1 正常模式（80%+ 回合）

```javascript
{
  temperature: 1.1,   // 更自然的表达
  maxTokens: 700,     // 短消息输出
  timeout: 15000      // 15秒超时
}
```

### 4.2 严谨模式（<20% 回合，关键场景）

```javascript
{
  temperature: 0.6,   // 更严谨准确
  maxTokens: 1200,    // 更完整输出
  timeout: 20000      // 20秒超时
}
```

**严谨模式触发场景**：
1. 新病例开场（定议程、分工、追问）
2. AI 长文消化（提炼争点、分工、追问）
3. 阶段总结（每 8 轮一次）
4. 讨论失控（开放问题 >3 个）

### 4.3 为什么不用 deepseek-reasoner？

- `deepseek-reasoner` 不支持结构化 JSON 输出
- reasoner 的 `content` 字段为空，推理过程在 `reasoning_content`
- 不适合需要 JSON segments 的结构化输出场景
- 使用 `deepseek-chat` + 参数优化更稳定、更省 token

## 5. 上下文管理

### 5.1 上下文裁剪策略

每次 API 调用只包含必要信息，避免 token 浪费：

**包含**：
- 最近 15 条对话（裁剪到 2000 字）
- 2~6 条关键事实（带数值/检查名）
- 1 条主矛盾
- 1~3 条未闭环问题
- ≤6 条课堂记忆要点

**不包含**：
- 完整病例全文（每轮都塞）
- 完整 transcript
- debug JSON 混入正文

### 5.2 Messages 结构

```javascript
[
  { role: 'system', content: systemPrompt },        // 统一约束
  { role: 'developer', content: outputProtocol },   // 输出协议
  { role: 'developer', content: roleCard },         // 本轮角色规则卡
  // ... 裁剪后的对话历史
]
```

## 6. 自定义 AI 角色

### 6.1 修改角色规则卡

编辑 `ai_roles/` 目录下的 Markdown 文件即可调整角色行为：

- `AI_助教团总规则.md`：修改所有角色的总体行为规范
- `AI_触发路由规则.md`：调整触发条件和角色选择逻辑
- `AI_兜底回复模板.md`：自定义异常情况下的回复

### 6.2 调整模型参数

模型参数在代码模块中配置：

- PBL2：`modules/pbl2/deepseek/deepseekClient.js`
- PBL3：`modules/pbl3/deepseek/deepseekClient.js`

修改 `chatParams` 和 `rigorousParams` 对象即可调整温度和 token 限制。

## 7. 故障排查

### 7.1 AI 不回复

1. 检查 `.env` 中 `DEEPSEEK_API_KEY` 是否正确
2. 确认 API 账户余额充足
3. 查看触发条件是否满足（参考第 3 节）
4. 检查服务器日志中的 API 调用错误

### 7.2 回复质量差

1. 检查角色规则卡是否被误修改
2. 调整 `temperature` 参数（降低可提高确定性）
3. 增加 `maxTokens` 以获得更完整回复
4. 检查上下文裁剪是否过于激进

### 7.3 API 超时

1. 检查网络连通性：`curl https://api.deepseek.com`
2. 增加 `timeout` 参数值
3. 考虑使用国内 API 代理加速
